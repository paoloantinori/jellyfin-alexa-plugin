## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Add coarse-grained media type toggles to `PluginConfiguration` that control which categories of content Alexa can access. This is the simplest layer of content access control â€” a global on/off switch per media type.

**Media type toggles to add:**
- `MusicEnabled` (bool, default: true) â€” controls access to Audio, MusicAlbum, MusicArtist items
- `VideosEnabled` (bool, default: true) â€” controls access to Movie, Episode items  
- `PodcastsEnabled` (bool, default: true) â€” controls access to podcast Series and Episodes
- `LiveTvEnabled` (bool, default: true) â€” controls access to LiveTvChannel items
- `BooksEnabled` (bool, default: true) â€” controls access to audiobooks/Book items

**Implementation**: Add booleans to `PluginConfiguration`. Create a `ContentAccessService` (or extend an existing helper) that all handlers call before querying Jellyfin â€” it injects `IncludeItemTypes` and `MediaType` filters based on enabled toggles. The config page gets a "Media Type Access" section with 5 toggle switches.

**Why**: This is the most common request â€” "I only use Alexa for music, don't show my movies." A single toggle per type covers 90% of use cases before needing collection-level granularity.

**ContentAccessService** â€” create a new service class that handlers call to get allowed `BaseItemKind[]` and `MediaType[]` filters:
```csharp
public class ContentAccessService
{
    public IReadOnlyList<BaseItemKind> GetAllowedItemKinds(PluginConfiguration config);
```

**Handler integration points:**
- `BaseHandler` gets a helper method `GetContentTypeFilter()` that returns Jellyfin query parameters based on enabled media types
- Each search/play handler injects these filters into their `ItemsClient.GetItemsAsync()` calls
- Handlers whose entire media type is disabled (e.g., PlayChannelIntent with LiveTvEnabled=false) return a localized "this content type is not available" response

**Note**: `PodcastsEnabled` here overlaps with the feature flag `PodcastsEnabled` in JF-134.1. They serve different purposes: JF-134.1 disables the intent handler entirely; JF-135.1 removes podcast content from search results and library queries. When either is false, podcasts should be inaccessible. The effective check is `config.PodcastsEnabled && config.PodcastsEnabledFeatureFlag` â€” or we could use a single flag for both. Decide during implementation whether to unify or keep separate.
