## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Add boolean feature flags to `PluginConfiguration` that allow administrators to enable/disable entire intent groups. When a flag is off, the corresponding handlers should return a polite "not available" response rather than executing.

**Feature flag groups to add:**
- `RadioModeEnabled` — controls PlayRadioIntent, TurnRadioOn/Off (default: true)
- `PodcastsEnabled` — controls PlayPodcastIntent (default: true)
- `LiveTvEnabled` — controls PlayChannelIntent (default: true)
- `SleepTimerEnabled` — controls SleepTimerIntent (default: true)
- `QueueManagementEnabled` — controls AddToQueue, PlayNext, ClearQueue, ListQueue (default: true)
- `BrowseLibraryEnabled` — controls BrowseLibraryIntent (default: true)
- `RecommendationsEnabled` — controls RecommendIntent (default: true)
- `AplVisualsEnabled` — controls APL template rendering on Echo Show/Fire TV (default: true)
- `VideoPlaybackEnabled` — controls PlayVideoIntent, PlayEpisodeIntent (default: true)

**Implementation approach**: Add a `Dictionary<string, bool>` or individual booleans to `PluginConfiguration`. In the config page, render a "Feature Flags" section with toggle switches. Each handler's `CanHandle()` or early `HandleAsync()` checks its flag and returns a localized "this feature is not enabled" response when off.

**Why**: Not every Jellyfin instance has live TV, podcasts, or even video content. Feature flags let admins tailor the Alexa experience to their library and avoid confusing "not found" errors for features they don't use.

## Implementation Details

**Config model** — add individual booleans to `PluginConfiguration` (not a dictionary — simpler serialization, clearer defaults):
```csharp
public bool RadioModeEnabled { get; set; } = true;
public bool PodcastsEnabled { get; set; } = true;
public bool LiveTvEnabled { get; set; } = true;
public bool SleepTimerEnabled { get; set; } = true;
public bool QueueManagementEnabled { get; set; } = true;
public bool BrowseLibraryEnabled { get; set; } = true;
public bool RecommendationsEnabled { get; set; } = true;
public bool AplVisualsEnabled { get; set; } = true;
public bool VideoPlaybackEnabled { get; set; } = true;
```

**Config page** — use `emby-collapse` section with `checkboxContainer-withDescription` for each flag:
```html
<div is="emby-collapse" title="Feature Flags">
  <div class="checkboxContainer checkboxContainer-withDescription">
    <label><input type="checkbox" is="emby-checkbox" id="RadioModeEnabled" /> Radio Mode</label>
    <div class="fieldDescription checkboxFieldDescription">Enable radio mode with auto-similar tracks</div>
  </div>
  <!-- repeat for each flag -->
</div>
```

**Handler gating** — two approaches, pick one:
1. **`CanHandle()` check**: Return false when flag is off → falls through to FallbackIntentHandler. Simpler but gives generic "I don't understand" response.
2. **Early `HandleAsync()` check**: Handler matches but returns localized "this feature is not enabled" response. Better UX since the user gets a clear explanation.

Recommend approach #2. Create a shared helper method `IsFeatureEnabled(string feature)` on `BaseHandler` that checks the config and returns a standardized response when disabled.

**Interaction model consideration**: When flags are disabled, consider whether to exclude corresponding sample utterances from SMAPI model updates. This would prevent Alexa from even routing to disabled intents. However, this adds complexity to `UpdateInteractionModelAsync`. Start with handler-level gating; model exclusion can be a follow-up.
