## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Extend the existing per-user skill configuration (already in `PluginConfiguration.Users`) to include content access overrides. This allows different household members to see different media through Alexa.

**Per-user content access properties to add to the User model:**
- `AllowedMediaTypes` (List<string>, optional) — overrides global media type toggles for this user (e.g., only music)
- `AllowedCollections` (List<string>, optional) — overrides global collection permissions for this user
- `MaxParentalRating` (int?, optional) — maximum content parental rating this user can access via Alexa

**Merge rule**: User permissions are always a subset of global permissions. If global denies movies, a user cannot override to allow movies. The effective permission is the intersection of global and per-user settings. If per-user is not configured, global settings apply.

**Implementation**: Extend the `User` class in the plugin configuration. The `ContentAccessService` (from JF-135.1/135.2) takes a userId parameter and resolves effective permissions by intersecting global + user settings. The config page user table gets an expandable row with content access checkboxes.

**Why**: A family may have adults and children using the same Jellyfin server with different Alexa devices. The parent's Echo can access everything; the kids' Echo is restricted to children's content.

## Implementation Notes

<!-- SECTION:NOTES:BEGIN -->
## Codebase Findings

**Per-user infrastructure already exists:**
- `PluginConfiguration.Users` collection holds per-user configs
- `Entities/User.cs` has properties: `JellyfinUserId`, `JellyfinToken`, `AlexaPersonId`, `ProactiveEventsEnabled`
- `BaseHandler.HandleRequestAsync` resolves user via PersonId or AccessToken → `_config.GetUserByPersonId()` / `_config.GetUserById()`

**Implementation approach**: Add `AllowedLibraryIds` and `DeniedLibraryIds` string arrays to the `User` entity. When building queries, resolve the user first, then inject their library restrictions.

**Challenge**: The current `User` entity is a flat DTO. Adding collection IDs requires updating:
1. `Entities/User.cs` — add collection ID arrays
2. `Configuration/config.html` — extend user table rows with library selector dropdown
3. `BaseHandler.cs` — add `GetUserCollectionFilter()` method that reads user-specific library IDs
4. Search/browse handlers — call user collection filter after resolving user

**Alternative simpler approach**: Instead of per-user library IDs, add per-user media type overrides (e.g., "Paolo can access music and podcasts, Maria can access everything"). This reuses the existing media type toggle infrastructure from JF-135.1 but at per-user level.

**Key insight**: Per-user rules require the user to be resolved BEFORE query construction. Currently `HandleRequestAsync` resolves the user (line 158-181), then calls `HandleAsync` with the resolved user. So `HandleAsync` already has the user context needed for per-user filtering.
